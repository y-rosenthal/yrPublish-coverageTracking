2  Coverage Tracking Spec

2.1 Goal

Provide a portable coverage-tracking add-on for Quarto books so students can see what has been covered and professors can compare sections, edit coverage, and maintain data in human-friendly files.

The implementation is static-site friendly (GitHub Pages compatible) and easy to move into a larger yrPublish feature collection.

2.2 Scope

2.2.1 In scope

  • Section-level coverage keyed by page URL + HTML section ID.
  • Student and professor modes in one JavaScript/CSS add-on.
  • CSV-based coverage data (one file per rendered HTML page per class section; see file layout below).
  • Professor UI tools for selecting coverage and exporting CSV text.
  • Helper scripts for adding missing heading IDs and updating coverage CSVs.

2.2.2 Out of scope

  • Auth and server-side write APIs.
  • Hosted database dependency.
  • Real-time writes from browser directly into repository files.

2.3 Functional requirements

2.3.1 FR-1: Coverage identity

  • Coverage is keyed by:
    • URL (for example /chapter1.html)
    • HTML ID (for example sec-limits-review)
    • Class section ID (for example MATH101-01)

2.3.2 FR-2: Student section picker

  • On first visit, students must choose exactly one class section.
  • Content is blurred until selection is made.
  • Student selection is stored per browser tab/session (sessionStorage).
  • Deep links must still trigger picker gating when no section is selected.

2.3.3 FR-3: Student highlighting

  • Covered items are highlighted.
  • Missing coverage entry defaults to not covered.
  • Sidebar control shows selected section name and lets student reset/re-pick.

2.3.4 FR-4: Professor mode and navigation

  • Professor mode activates with ?prof or ?prof=true.
  • Professor chooser supports selecting zero or more sections using checkboxes.
  • Chooser is not auto-opened on navigation; opens when professor clicks sidebar controls.
  • Esc dismisses chooser without applying changes.
  • Internal links preserve prof query parameter.

2.3.5 FR-5: Professor sidebar controls

  • Sidebar top shows selected professor sections as color-coded entries.
  • Clicking any selected section entry re-opens chooser.
  • If none selected, show Choose section(s) entry.
  • Include Edit page coverage CSV entry to open CSV editor.

2.3.6 FR-5b: Professor inline coverage toggles

  • When professor mode is active and at least one class section is selected, the tracker shows per-section checkboxes next to each trackable heading (or chapter title span) in the page content, color-coded by section.
  • Toggling an inline checkbox updates in-memory coverage, refreshes highlights, and stays in sync with the CSV editor dialog when it is open.
  • Nested ids are logical overrides: if an outer tracked id is covered but a nested tracked id is not, the nested region is rendered as uncovered.
  • Outer-id toggles may cascade to nested ids:
    • checking outer id: auto-check all descendants when all are unchecked; for mixed descendants, show custom choice dialog (Check this section and all nested sections vs Only check this section)
    • unchecking outer id: auto-uncheck descendants when all checked or all unchecked; for mixed descendants, show custom choice dialog (Uncheck this section and all nested sections vs Only uncheck this section)
    • pressing Esc in the nested decision dialog cancels and applies no checkbox changes
  • Inline checkbox actions support undo/redo hotkeys in professor mode (Ctrl+Z / Ctrl+Y, plus macOS Cmd+Z / Cmd+Shift+Z).

2.3.7 FR-6: Coverage data file format

  • coverage-config.json remains JSON and defines sections and colors (optional coverageDataDir per section).
  • Coverage data is stored under coverage-data/<SECTION-ID>/ with one CSV per HTML page: yr-coverage--<page-slug>.csv (see proposed file structure). The runtime loads only the current page’s files.
  • CSV row formats:
    • URL,html-id,covered
    • URL,html-id,not-covered
    • URL,html-id (defaults to covered)
    • URL,html-id, (defaults to covered)
  • If (URL,html-id) is missing from the relevant page’s CSV for that section, default is not covered.

2.3.8 FR-7: Professor CSV editing UI

  • Professor can open page-level coverage editor in the UI.
  • Editor supports:
    • a single multi-line text-editor-style control for all rows on the page
    • one visible CSV row per trackable html-id, with the checkbox aligned in a gutter directly next to that row text
    • tri-state row status cycle: covered -> explicit not-covered -> implicit not-covered (omitted from CSV output)
    • set all page IDs to covered
    • set all page IDs to explicit not-covered
    • set all page IDs to implicit not-covered
    • convert all not-covered rows to explicit form
    • convert all not-covered rows to implicit form
    • undo/redo editor history (buttons + keyboard shortcuts Ctrl+Z / Ctrl+Y, recognized while the dialog is open)
    • copy CSV text
    • download CSV file (filename aligned with yr-coverage--<page-slug>.csv naming)
  • Implicit rows are visually grey/italic with a greyed checkbox, but still visible so users can promote them to explicit rows.
  • Manual edits to a row make it explicit; deleting/emptying a row line reverts that row to implicit not-covered (grey/italic) rather than removing it from the editor surface.
  • The dialog is draggable and non-modal; underlying page content remains scrollable while editing, and it is closed via an upper-right × button.
  • Workflow remains static-first: professor can copy/download and save offline into the corresponding repo file.

2.3.9 FR-7b: Trackable HTML ids (idsOnPage)

  • The runtime collects the same set of ids for highlighting, inline toggles, and the editor. In Quarto HTML books this typically includes elements with ids on section wrappers and chapter title h1 span[id] nodes, not only h1[id] / h2[id] on the heading element itself.

2.3.10 FR-8: Helper scripts

  • tools/yrAddSectionsIds.sh
    • input: one .qmd file
    • output: new .qmd with missing heading IDs inserted
    • skips fenced code blocks so # inside code is ignored
  • tools/yrUpdateCoverage.sh
    • input: section id, book root, data dir, one or more .qmd files
    • output: per-page yr-coverage--<page-slug>.updated.csv next to each target yr-coverage--<page-slug>.csv
    • adds missing (URL,html-id) rows as not-covered

2.4 Proposed file structure

assets/coverage-tracking/
  coverage-config.json
  coverage-data/
    <SECTION-ID>/
      yr-coverage--index.html.csv
      yr-coverage--chapter1.html.csv
      yr-coverage--part-a--chapter.html.csv
  coverage-tracker.js
  coverage-tracker.css
  coverage-tracker.html
tools/
  yrAddSectionsIds.sh
  yrUpdateCoverage.sh

2.5 CSV semantics

Example section file:

URL,html-id,covered
/index.html,sec-how-to-use,covered
/index.html,sec-reading-plan,not-covered
/chapter1.html,sec-limits-review,

Interpretation:

  • Third column covered => covered
  • Third column not-covered => not covered
  • Missing/blank third column => covered
  • Missing row entirely => not covered

2.6 Sample demo book

The sample implementation is in sampleBooks/sample1/ and demonstrates:

  • student picker and highlighting
  • professor compare mode
  • professor sidebar controls
  • professor inline per-section checkboxes in page content (when at least one section is selected)
  • professor CSV editor/export workflow (tri-state per-line rows and inline CSV text editing)
  • per-page CSV coverage loading (lazy fetch per page)

2.7 Risks and mitigations

  • Missing stable section IDs:
    • mitigate with yrAddSectionsIds.sh and explicit {#id} authoring practice.
  • Inconsistent URLs between source and rendered pages:
    • mitigate with conventions in tooling and editor guide.
  • Theme collisions:
    • mitigate with namespaced .ct-* CSS selectors.

2.8 Definition of done

  • Coverage runtime reads coverage-config.json + per-page CSV files for the current section(s).
  • Student mode enforces picker gating and tab-scoped persistence.
  • Professor mode supports comparison, navigation persistence, sidebar controls, and chooser dismiss-on-Esc.
  • Professor can generate/export CSV coverage from UI.
  • Tools scripts exist and are documented.
  • Docs include setup and operations guide for editors/professors.
