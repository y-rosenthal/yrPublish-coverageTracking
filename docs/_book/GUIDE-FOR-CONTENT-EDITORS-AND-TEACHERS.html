4  Guide for Content Editors and Teachers

4.1 Who this guide is for

This guide is for people who maintain Quarto books and track class coverage.

Students do not need special documentation to consume the book.

4.2 What this system does

  • Students pick a class section and see covered content highlighted.
  • Professors can compare multiple sections in ?prof mode.
  • Coverage data is maintained in per-class-section directories, with one CSV per rendered HTML page (see naming below).
  • Professors can generate updated CSV content from the website UI.

4.3 Required directory structure

In your Quarto book project, include:

assets/coverage-tracking/
  coverage-config.json
  coverage-data/
    <SECTION-ID>/
      yr-coverage--index.html.csv
      yr-coverage--chapter1.html.csv
      yr-coverage--part-a--chapter.html.csv
  coverage-tracker.js
  coverage-tracker.css
  coverage-tracker.html

Filename rule: yr-coverage-- + slug + .csv, where slug is the page path without a leading slash, with / replaced by -- (e.g. /chapter1.html → chapter1.html; /part-a/chapter.html → part-a--chapter.html).

Optional helper scripts:

tools/
  yrAddSectionsIds.sh
  yrUpdateCoverage.sh

4.4 File formats

4.4.1 coverage-config.json (JSON, required)

Defines available sections and styling.

{
  "sections": [
    {
      "id": "MATH101-01",
      "label": "MATH101 Section 01",
      "color": "#2e7d32"
    }
  ]
}

Optional: "coverageDataDir": "assets/coverage-tracking/coverage-data/MATH101-01/" if the section directory is not the default coverage-data/<SECTION-ID>/.

4.4.2 coverage-data/<SECTION-ID>/yr-coverage--<page-slug>.csv (CSV, required per page you track)

Each class section has its own directory. Each rendered page you care about has its own CSV file in that directory. Large books stay maintainable: you edit only the files for the pages you change.

Accepted row formats:

  • URL,html-id,covered
  • URL,html-id,not-covered
  • URL,html-id
  • URL,html-id,

Default semantics:

  • Missing row => not covered
  • Blank/missing third column => covered

Example:

URL,html-id,covered
/chapter1.html,sec-limits-review,covered
/chapter1.html,sec-continuity-basics,not-covered
/chapter1.html,sec-derivatives-preview,

4.5 Quarto integration steps

In your book _quarto.yml:

  1. Include coverage config and CSV directory as project resources.
  2. Include coverage CSS.
  3. Include the tracker script HTML include.

Example:

project:
  type: book
  resources:
    - assets/coverage-tracking/coverage-config.json
    - assets/coverage-tracking/coverage-data/

format:
  html:
    css:
      - assets/coverage-tracking/coverage-tracker.css
    include-after-body:
      - assets/coverage-tracking/coverage-tracker.html

4.6 Authoring content safely

Coverage is based on HTML IDs, so section headings should have explicit IDs:

## Limits Review {#sec-limits-review}

This makes coverage stable and portable across edits.

4.7 Helper tools

4.7.1 yrAddSectionsIds.sh

Purpose:

  • add missing heading IDs in a .qmd file
  • skip fenced code blocks

Usage:

./tools/yrAddSectionsIds.sh input.qmd [output.qmd]

4.7.2 yrUpdateCoverage.sh

Purpose:

  • For a given class section, update one per-page CSV per .qmd file.
  • Adds missing (URL,html-id) rows as not-covered.
  • Writes yr-coverage--<page-slug>.updated.csv next to each target yr-coverage--<page-slug>.csv.

Usage (from your Quarto book root, so URLs resolve to /index.html, /chapter1.html, etc.):

./tools/yrUpdateCoverage.sh --section MATH101-01 --data-dir assets/coverage-tracking/coverage-data index.qmd chapter1.qmd

From another working directory, pass --book-root pointing at the book project:

./tools/yrUpdateCoverage.sh --section MATH101-01 --book-root path/to/book --data-dir path/to/book/assets/coverage-tracking/coverage-data path/to/book/index.qmd

4.8 Professor workflow in UI

4.8.1 Compare mode

  1. Open book with ?prof or ?prof=true.
  2. Use sidebar entry to open section chooser.
  3. Select zero or more sections with checkboxes.
  4. Apply and review highlighted overlap.

4.8.2 Inline toggles (in page content)

  1. After at least one class section is selected, each trackable heading on the page shows small checkboxes (one per selected section), color-coded by section.
  2. Toggling an inline checkbox updates highlights immediately and stays consistent with the CSV editor dialog if it is open.
  3. Nested IDs are respected:
    • if an outer id is covered but a nested tracked id is not, the nested region is rendered as uncovered
    • checking/unchecking an outer id can cascade to nested ids
    • for mixed nested states, a custom draggable decision dialog appears with explicit choices (for example, Check this section and all nested sections vs Only check this section)
    • pressing Esc on that decision dialog cancels and applies no checkbox changes
  4. Outside the editor dialog, professor inline checkbox actions support undo/redo hotkeys (Ctrl+Z / Ctrl+Y, and Cmd+Z / Cmd+Shift+Z on macOS).

4.8.3 Edit page coverage CSV

  1. In professor mode, click Edit page coverage CSV in sidebar.
  2. Choose the class section to edit.
  3. The dialog uses a single multi-line text-editor-style control. Each HTML id maps to one row, with a checkbox aligned in a left gutter next to that row text.
  4. Rows are indented by 2 spaces per nesting level so parent/child structure is visible in the editor view (indentation is visual only and is not written to exported CSV files).
  5. Checkbox state is tri-state per row:
    • checked = explicit covered
    • unchecked = explicit not-covered
    • grey/italic (indeterminate) = implicit not-covered (line is omitted from CSV output)
  6. Use:
    • Undo / Redo (or keyboard Ctrl+Z / Ctrl+Y; Cmd+Z / Cmd+Shift+Z on macOS)
    • Set all covered
    • Set all explicit not-covered
    • Set all implicit not-covered
    • Make all not-covereds explicit
    • Make all not-covereds implicit
    • individual row toggles (cycle through covered → explicit not-covered → implicit not-covered)
  7. Dialog checkbox toggles stay in sync with inline professor checkboxes shown in the page text, and nested-toggle prompts/cascade rules are consistent in both places.
  8. If a line is edited to empty/deleted, that row does not disappear; it reverts to implicit not-covered (grey/italic).
  9. You can manually edit any row text. If you start editing an implicit (grey) row, it becomes explicit (not grey) so that row is included in output.
  10. The dialog is draggable and non-modal, so you can move it and scroll/read the underlying page while editing. Use the × button in the top-right of the dialog to close it.
  11. Undo/redo hotkeys are recognized while the dialog is open (not only when the text area is focused).
  12. Copy / Download exports only explicit rows (plus header). Save/update the matching yr-coverage--<page-slug>.csv file in the repo under coverage-data/<SECTION-ID>/.

4.8.4 Heading IDs in Quarto HTML

Quarto often places the stable anchor on a <section id="..."> wrapper (with the visible heading inside), or on a <span id="..."> inside the chapter title h1. Use explicit {#id} on headings in source when possible, and ensure CSV html-id values match the rendered DOM ids (see the sample book).

4.9 Notes on defaults and new content

  • If a new heading is added but not listed in CSV, it is interpreted as not covered.
  • This default is useful during course progression because coverage accumulates over time.

4.10 Moving this feature into another repository

To copy this feature into a larger Quarto feature collection:

  1. Copy assets/coverage-tracking/.
  2. Copy tools/ scripts if desired.
  3. Update target book _quarto.yml.
  4. Render and test student + professor modes.
